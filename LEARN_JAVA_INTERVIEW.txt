[ ] Q: В чём разница между == и equals()?
    A: == сравнивает ССЫЛКИ (один ли объект в памяти).
       equals() сравнивает СОДЕРЖИМОЕ (переопределён в String, Integer и т.д.)
       "abc" == "abc" → зависит от пула строк (string pool)
       "abc".equals("abc") → всегда true
       Правило: для объектов всегда используй equals()!

[ ] Q: Что такое String Pool?
    A: Область памяти (heap) где JVM хранит строковые литералы.
       String a = "hello"; String b = "hello"; → a == b (один объект в пуле!)
       String a = new String("hello"); → НОВЫЙ объект, не в пуле
       intern() — принудительно поместить в пул

[ ] Q: В чём разница int и Integer?
    A: int — примитив (8 байт в стеке, нет методов, не может быть null)
       Integer — объект-обёртка (reference type, есть методы, может быть null)
       Autoboxing: int → Integer (автоматически)
       Unboxing: Integer → int (автоматически)
       ОСТОРОЖНО: Integer.valueOf(127) == Integer.valueOf(127) → true (кеш -128..127)
                  Integer.valueOf(200) == Integer.valueOf(200) → false!

[ ] Q: Что такое immutable класс? Как создать?
    A: Объект, состояние которого нельзя изменить после создания.
       String, Integer, LocalDate — все immutable.
       Как создать:
         1. final class (нельзя наследовать)
         2. Все поля private final
         3. Нет сеттеров
         4. Конструктор глубоко копирует мутабельные поля
         5. Геттеры возвращают копии мутабельных объектов

[ ] Q: В чём разница StringBuilder и String?
    A: String — immutable. Каждая конкатенация создаёт НОВЫЙ объект!
       "a" + "b" + "c" → создаёт 2 временных строки
       StringBuilder — mutable. append() меняет ОДИН объект.
       В цикле ВСЕГДА используй StringBuilder!
       StringBuffer — как StringBuilder но потокобезопасный (медленнее, редко нужен)

[ ] Q: Что такое final, finally, finalize?
    A: final — модификатор (переменная = константа, метод = нельзя переопределить,
               класс = нельзя наследовать)
       finally — блок try/catch/finally (выполняется ВСЕГДА, даже при исключении)
       finalize() — метод Object, вызывается GC перед удалением (deprecated, не используй!)

[ ] Q: Объясни Stack и Heap в JVM
    A: Stack — память для стека вызовов (local variables, primitives, references)
               LIFO, быстрый, для каждого потока свой
               StackOverflowError при бесконечной рекурсии
       Heap — память для объектов (new Object())
               Общая для всех потоков, управляется GC
               OutOfMemoryError когда нет места

[ ] Q: 4 принципа ООП?
    A: Инкапсуляция — скрытие данных (private поля + public методы)
       Наследование — extends, код переиспользуется (DebitAccount extends Account)
       Полиморфизм — разное поведение одного метода (withdraw() у Debit vs Credit)
       Абстракция  — скрываем реализацию, показываем интерфейс (abstract, interface)

[ ] Q: abstract class vs interface?
    A: abstract class:
         - Может иметь поля, конструкторы
         - Может иметь реализованные методы
         - Одиночное наследование (extends только один)
         - Используй когда: есть общее состояние + частичная реализация
       interface:
         - Только абстрактные методы (до Java 8)
         - С Java 8: default и static методы
         - Множественная реализация (implements много)
         - Используй когда: определяешь контракт поведения
       Правило: "is-a" → abstract class. "can-do" → interface
       Car is-a Vehicle (extends). Car can-do Flyable (implements)

[ ] Q: Что такое полиморфизм? Приведи пример.
    A: Один и тот же метод работает по-разному для разных типов.
       Account acc = new SavingsAccount(...); // переменная типа родителя
       acc.withdraw(1000); // вызовет метод ПОДКЛАССА (динамическое связывание)

       Это называется "upcasting" — SavingsAccount IS-A Account.
       Обратное (downcasting): SavingsAccount sa = (SavingsAccount) acc;
       Проверка перед downcasting: if (acc instanceof SavingsAccount sa) { ... }

[ ] Q: Зачем нужен equals() и hashCode()? Почему вместе?
    A: equals() — проверяет логическое равенство объектов
       hashCode() — возвращает числовой хеш (используется в HashMap, HashSet)

       Контракт: если a.equals(b) → a.hashCode() == b.hashCode()
       (Обратное не обязательно)

       Если переопределить только equals() — HashMap/HashSet будет работать НЕПРАВИЛЬНО!
       (Объекты будут класться в разные buckets и не будут "находиться")

[ ] Q: Что такое SOLID? Объясни на примере банка.
    A: S — Single Responsibility: класс делает одно дело
           AccountService только бизнес-логика, не работает с БД напрямую
       O — Open/Closed: открыт для расширения, закрыт для изменения
           Новый тип счёта → новый класс extends Account, не меняем Account
       L — Liskov Substitution: подкласс заменяет родителя
           Везде где Account — можно использовать DebitAccount
       I — Interface Segregation: много маленьких интерфейсов
           Transferable отдельно, Auditable отдельно (не один огромный IAccount)
       D — Dependency Inversion: зависеть от абстракций, не конкретных классов
           AccountService зависит от AccountRepository (интерфейс), не от PostgresRepository

[ ] Q: ArrayList vs LinkedList — когда использовать?
    A: ArrayList:
         - O(1) доступ по индексу (get(i))
         - O(n) вставка/удаление в середине
         - Лучше для: чтение, iteration
         - Использует массив внутри
       LinkedList:
         - O(n) доступ по индексу
         - O(1) вставка/удаление в начале/конце
         - Лучше для: частые добавления в начало
         - Больше памяти (ссылки prev, next)
       В 95% случаев → ArrayList!

[ ] Q: HashMap — как работает внутри?
    A: 1. Вычисляем hashCode() ключа
       2. Определяем bucket (index = hash & (capacity-1))
       3. Кладём в bucket (если пустой — просто, если нет — chaining)

       Java 8+: если цепочка > 8 элементов → TreeMap (O(log n) вместо O(n))

       Capacity (ёмкость) = 16 по умолчанию
       Load factor = 0.75 (при заполнении 75% — resize × 2)

       Идеальный ключ: неизменяемый (String, Integer — лучшие ключи)
       Плохой ключ: мутабельный объект (можно изменить после вставки → потеря!)

[ ] Q: В чём разница HashMap, LinkedHashMap, TreeMap?
    A: HashMap — O(1) операции, НЕТ гарантии порядка
       LinkedHashMap — O(1) операции, сохраняет порядок ВСТАВКИ
       TreeMap — O(log n) операции, сортировка по ключу (Comparable)

       Выбор:
         Нужен порядок вставки → LinkedHashMap
         Нужна сортировка → TreeMap
         Нужна скорость → HashMap

[ ] Q: Что такое ConcurrentModificationException?
    A: Изменение коллекции во время итерации по ней.
       НЕПРАВИЛЬНО:
         for (String s : list) { if (s.equals("X")) list.remove(s); } // CME!
       ПРАВИЛЬНО:
         Iterator<String> it = list.iterator();
         while (it.hasNext()) { if (it.next().equals("X")) it.remove(); }
         // или: list.removeIf(s -> s.equals("X"));

[ ] Q: Checked vs Unchecked исключения?
    A: Checked — надо объявить в throws или поймать в try/catch
         IOException, SQLException — их НАДО обрабатывать
         Компилятор проверяет
       Unchecked (RuntimeException и его подклассы) — не обязательно ловить
         NullPointerException, IllegalArgumentException, IllegalStateException

       В банке: InsufficientFundsException → Unchecked (RuntimeException)
       (Потому что это бизнес-ошибка, которую сложно "восстановить")

[ ] Q: finally vs try-with-resources?
    A: finally выполняется ВСЕГДА:
         try { ... } catch (Exception e) { ... } finally { resource.close(); }

       try-with-resources (Java 7+) — ресурс закрывается автоматически:
         try (Connection conn = getConnection()) { ... }
         // conn.close() вызывается автоматически даже при исключении
         Ресурс должен реализовывать AutoCloseable/Closeable

[ ] Q: Что такое race condition? Пример.
    A: Два потока читают и изменяют одно значение одновременно.
       balance = 100;
       Thread 1: read(100) → Thread 2: read(100) → T1: write(90) → T2: write(90)
       Итог: balance=90 вместо 80! (потеряли одно снятие)

       Решение: synchronized, AtomicInteger, ReentrantLock

[ ] Q: volatile vs synchronized?
    A: volatile — гарантирует видимость изменений между потоками
                  (без кеширования в thread-local cache)
                  НЕ атомарность! volatile int i; i++ — НЕ потокобезопасен!
       synchronized — гарантирует видимость И атомарность (один поток в методе)
                      НО: медленнее (блокировка)

       Правило: для флагов → volatile. Для сложных операций → synchronized или Lock.

[ ] Q: Что такое deadlock? Как избежать?
    A: Два потока ждут ресурсы которые держит другой → вечное ожидание.
       T1 захватил A, ждёт B.
       T2 захватил B, ждёт A. → Deadlock!

       Как избежать:
         1. Всегда захватывать блокировки В ОДНОМ ПОРЯДКЕ
         2. Использовать tryLock() с таймаутом
         3. Избегать вложенных synchronized

[ ] Q: ExecutorService vs new Thread()? Зачем нужен пул потоков?
    A: new Thread() — создавать поток дорого (стек 512KB+, OS операция)
       1000 запросов → 1000 потоков → OutOfMemoryError!

       ExecutorService (пул потоков):
         - N потоков создаются один раз
         - Задачи ставятся в очередь
         - Потоки переиспользуются
         - Контроль количества потоков

       В Spring Boot: уже встроен (Tomcat thread pool, @Async, @Scheduled)

[ ] Q: CompletableFuture — зачем?
    A: Асинхронное выполнение без блокировки потока.
       // Синхронно (блокирует поток):
       String result = callSlowService(); // ждём здесь

       // Асинхронно:
       CompletableFuture<String> future = CompletableFuture.supplyAsync(() -> callSlowService());
       future.thenApply(r -> r.toUpperCase())  // когда готово — преобразуй
             .thenAccept(r -> System.out.println(r)); // и выведи
       // Основной поток не блокируется!

[ ] Q: Что такое IoC и Dependency Injection?
    A: IoC (Inversion of Control) — управление зависимостями передаётся фреймворку.
       DI — фреймворк сам создаёт и передаёт зависимости.

       БЕЗ DI: AccountService service = new AccountService(new AccountRepository(...));
       С DI:   @Autowired AccountService service; // Spring сам создаёт!

       Преимущества: слабая связность, легко тестировать (подменять мок)

[ ] Q: @Transactional — что делает?
    A: Оборачивает метод в транзакцию базы данных.
       Если метод выбрасывает RuntimeException → rollback (откат всех изменений)
       Если метод завершился успешно → commit

       @Transactional(rollbackFor = Exception.class) — rollback даже для checked
       @Transactional(readOnly = true) — оптимизация для чтения

[ ] Q: N+1 проблема в JPA?
    A: Загрузка N сущностей = 1 запрос для списка + N запросов для каждого
       (Например: 100 клиентов + 100 запросов для счетов каждого = 101 запрос!)

       Решения:
         @EntityGraph — явная загрузка связей
         JOIN FETCH в JPQL
         @BatchSize(size = 10) — загрузка пачками

[ ] Q: Индексы — зачем? Когда НЕ нужны?
    A: Индекс ускоряет поиск (O(log n) вместо O(n) для B-Tree индекса)
       CREATE INDEX idx_account_client ON accounts(client_id);

       Когда нужны:
         - WHERE поля (особенно по FK)
         - ORDER BY поля
         - JOIN поля

       Когда НЕ нужны:
         - Маленькие таблицы (< 1000 строк)
         - Поля с мало уникальных значений (gender — плохой индекс)
         - Таблицы с очень частыми UPDATE/INSERT (индексы замедляют запись!)

[ ] Q: Что такое транзакция? ACID?
    A: Транзакция — набор операций выполняемых атомарно.
       ACID:
         Atomicity   — всё или ничего (перевод: -1000 и +1000 вместе)
         Consistency — данные всегда в правильном состоянии (баланс ≥ 0)
         Isolation   — транзакции изолированы друг от друга
         Durability  — после commit данные сохранены (даже при сбое)

[ ] Q: Уровни изоляции транзакций?
    A: READ UNCOMMITTED — видит незакоммиченные изменения (dirty reads!)
       READ COMMITTED   — видит только закоммиченные (по умолчанию в PostgreSQL)
       REPEATABLE READ  — один и тот же SELECT возвращает одно (по умолчанию MySQL)
       SERIALIZABLE     — полная изоляция (самый медленный)

       Проблемы: dirty read, non-repeatable read, phantom read

[ ] Q: О-нотация — что это и зачем?
    A: Способ описать скорость алгоритма при росте входных данных.
       O(1)      — константно (получить элемент HashMap по ключу)
       O(log n)  — логарифм (бинарный поиск, BST операции)
       O(n)      — линейно (перебор списка)
       O(n log n) — быстрая сортировка в среднем, merge sort
       O(n²)     — пузырьковая сортировка (два вложенных цикла)
       O(2^n)    — перебор всех подмножеств (очень медленно!)

[ ] Q: Перевернуть строку?
    A: new StringBuilder(str).reverse().toString()
       Или:
       char[] chars = str.toCharArray();
       for (int i = 0, j = chars.length-1; i < j; i++, j--) {
           char tmp = chars[i]; chars[i] = chars[j]; chars[j] = tmp;
       }
       return new String(chars);

[ ] Q: Найти дубликаты в массиве?
    A: Set<Integer> seen = new HashSet<>();
       for (int num : arr) {
           if (!seen.add(num)) System.out.println("Дубликат: " + num);
       }
       // O(n) время, O(n) память

[ ] Q: Является ли строка анаграммой?
    A: "слово" и "влосо" — анаграммы (одни и те же буквы)
       char[] a = s1.toCharArray(); Arrays.sort(a);
       char[] b = s2.toCharArray(); Arrays.sort(b);
       return Arrays.equals(a, b); // O(n log n)

       Или через HashMap: считаем частоту букв для каждой строки
